Difference between SQA and SQC

  • SQA is oriented towards prevention, SQC is oriented towards detection

Verification vs. Validation

  • Verification: ensuring that a product is being built according to design specs. Are we building the product right?
  • Validation: ensuring that the product actually meets users’ needs, and that the specs were right in the first place. Are we building the right product?

Levels of Testing


Unit Testing

  • Individual units/components are tested.
  • A unit is the smallest testable part of the software, with usually one input and one output, usually a method or function
  • Unit testing frameworks, drivers, stubs and mock objects are often used
  • Method: White-box
  • Who: normally the developer or peer
  • Benefits:
    • increases confidence in changing/maintaining code
    • make codes more reusable
    • development is faster
    • cost of fixing defects is less
    • easy debugging
    • more reliable code

Integration Testing

  • Individual units are combined and tested as a group.
  • Purpose: to expose faults in the interaction between the integrated units
  • Method: any, Black, Gray or White Box
  • Approaches:
    • Big Bang: all or most of the units combined together and tested in one go
    • Top-down: top-level units are tested first. Test-stubs are needed to simulate lower level units
    • Bottom-up: bottom-level units are tested first. Test-drivers may be needed to simulate higher-level units
  • Who: Devs or testers

System Testing

  • The complete integrated software is tested.
  • Purpose: to evaluate the systems compliance with the specified requirements
  • Method: Black Box, structured
  • Who: Testers

Acceptance Testing

  • The system is tested for acceptability, and compliance with business requirements
  • Method: Black Box, ad-hoc
  • Who:
    • Internal (Alpha): members of the development organization, typically members of Product Management, Sales or Customer Support
    • External:
      • Customer Acceptance Testing
      • User Acceptance Testing



Methods of Testing


Black Box Testing

  • AKA Behavioral Testing
  • The internal structure of the system is not known to the tester
  • Usually functional
  • This method attempts to find errors in the following categories
    • incorrect or missing functions
    • interface errors
    • errors in data structures or external database access
    • behavior or performance errors
    • initialization and termination errors
  • Levels: integration, system, acceptance

Techniques

  • Equivalence Partitioning: dividing input values into valid and invalid categories and selecting representative values as test data
  • Boundary Value Analysis: selecting values that are just inside/outside of the boundaries as test data
  • Cause-Effect Graphing: producing a Cause-Effect graph by identifying the input and output conditions and generating the test cases accordingly

Advantages

  • Exposes discrepancies in the specs
  • No programming knowledge required
  • Tester group can be independent
  • Test cases can be designed ASA the specs are complete

Disadvantages

  • Small number of inputs can be tested
  • W/O clear specs test cases are difficult to design
  • Tests can be redundant

White Box Testing

  • The internal structure of the system is known to the tester
  • It is testing beyond the UI, into the nitty-gritty of the system
  • Levels: unit, integration, system

Advantages

  • Testing can begin early
  • Testing is more thorough

Disadvantages

  • Tests can be complex, programmer knowledge is required
  • Test script maintenance can be a burden
  • Tools may not be readily available

Gray Box Testing

  • Combination of White and Black Box
  • The internal structure of the software is partly known
  • Levels: integration

Ad-hoc Testing

  • AKA Monkey Test
  • Random input
  • levels: acceptance



Types of Tests


Smoke Test

  • AKA Build Verification Testing
  • Ensures that the most important functions work
  • It helps decide if a build is stable enough for further testing
  • Covers most of the major functions but none of them in depth
  • If builds are frequent, best to automate this test

Advantages

  • Exposes integration issues
  • Uncovers problems early
  • Provides confidence, that changes haven’t affected major areas

Functional Testing

  • The system is tested against functional requirements/specs
  • Functions are tested by feeding them input and examining the output
  • Inspects the output, not how it is generated
  • Method: Black Box
  • Levels: system and acceptance

Typical Steps

  • Identify the function the software is expected to perform
  • Create input data
  • Determine expected output
  • Execute
  • Compare

Usability Testing

  • Performed from the end-user’s perspective
  • Tests: how understandable, easy to use/operate, attractive the software is

Security Testing

  • Purpose: find vulnerabilities
  • Focus areas:
    • Network security
    • System software security
    • Client-side application security
    • Server-side application security

Performance Testing

  • Determines how a system performs under load
  • Test environment should be as close to production env. as possible
  • Test multiple times, and arrive at an average

Types

  • Load Testing: behavior at increasing workload
  • Stress Testing: behavior beyond the limits of anticipated workload
  • Endurance Testing: behavior under sustained heavy workload
  • Spike Testing: behavior when load is suddenly and substantially increased

Regression Testing

  • Ensures that changes had no adverse effect
  • Previously created test cases are re-executed
  • Levels: all, but most relevant during system testing

Compliance Testing

  • Determines the compliance of the system w/ internal or external standards



Testing Artifacts


Test Plan

  • A document describing the scope and activities of testing
  • Identifies:
    • the features to be tested
    • testing tasks
    • personnel
    • degree of tester independence
    • test environment
    • test design techniques
    • entry and exit criteria

Test Plan Types

  • Master Test Plan: high-level, unifies all other test plans
  • Testing Level Specific Test Plans:
    • Unit Test Plan
    • Integration Test Plan
    • System Test Plan
    • Acceptance Test Plan
  • Testing Type Specific Test: plans for major types of testing like Performance Test Plan and Security Test Plan

Defects


Defect / Bug

  • an error in coding or logic which produces incorrect results

Classification

  • Severity/Impact
  • Probability/Visibility
  • Priority/Urgency
  • Related Dimension of Quality
  • Related Module/Component
  • Phase Detected
  • Phase Injected

Defect Severity

  • The degree of impact a defect has on development or operation of a component/system

Classification

  • Critical: complete failure
  • Major: has difficult workaround
  • Minor: has easy workaround
  • Trivial: does not affect functionality or data

Defect Probability

  • Likelihood of a user encountering the bug
    • High: almost all users
    • Medium: about 50%
    • Low: very few

Defect Priority

  • The urgency of fixing a defect
  • Priority is subjective, the following will be considered:
    • Business need
    • Defect Severity
    • Defect Probability
    • Available Resources and Time

Classification

  • Urgent: needs to be fixed in the next build
  • High: needs to be fixed in upcoming builds
  • Medium: may be fixed after release / in the next release
  • Low: may or may not be fixed at all

Defect Life Cycle

  • The journey of a defect from identification to its closure

Status

  • New
  • Assigned/Open
  • Deferred
  • Dropped/Rejected
  • Completed/Fixed/Resolved/Test
  • Reassigned/Reopened
  • Closed/Verified

Defect Report

  • A document that describes a defect detected by the tester
  • It’s important to
    • Be
      • specific
      • detailed
      • objective
    • Reproduce the defect
    • Review the report

Metrics


Defect Age

  • Measured in:
    • Defect Age in Time = Defect Fix Date – Defect Detection Date
    • Defect Age in Phase = Defect Detection Phase – Defect Injection Phase

Defect Density

  • Size is measured in one of the following:
    • Function Points
    • Lines of Code

Defect Detection Efficiency

  • The closer to 100% is the better

Cost of Quality

  • Sums up the cost of prevention vs. occurence of defects
  • Made up of the following:
    • Cost of Control: prevention
    • Cost of Failure of Control
      • Internal Failure Cost: cost of defects identified internally and efforts to correct them
      • External Failure Cost: cost of defects identified by the client
